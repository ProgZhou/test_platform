# 测试方案

## 1. 测试范围

### 1.1 功能测试范围

| 模块 | 缩写 | 功能点覆盖 | 验收标准覆盖 |
|------|------|-----------|-------------|
| 测试用例管理 | CASE | 功能点 1-6 | AC-1 ~ AC-6 |
| 测试接口管理 | API | 功能点 7-12 | AC-7 ~ AC-12 |
| 测试脚本管理 | SCRIPT | 功能点 13-16 | AC-13 ~ AC-16 |
| 自动化执行 | EXEC | 功能点 17-18 | AC-17 ~ AC-18, AC-20 |
| 测试报告 | REPORT | 功能点 19-20 | AC-19 |

### 1.2 非功能测试范围

- 性能测试：页面操作响应时间、Excel 批量导入性能、接口调试超时、脚本执行超时
- 安全测试：沙箱隔离、文件上传校验、接口代理防跨域、SQL 注入 / XSS 防护
- 可用性测试：目录树拖拽排序、代码编辑器语法高亮、二次确认弹窗、模板下载
- 边界测试：8 个边缘场景全覆盖（Excel 格式错误、OpenAPI 解析失败、参数名校验、非空目录删除、不支持的脚本语言、脚本执行超时、并发执行冲突、大文件上传）

## 2. 测试策略

### 2.1 单元测试

| 维度 | 说明 |
|------|------|
| 前端框架 | React Testing Library + Vitest |
| 后端框架 | Go testing + testify |
| 覆盖率要求 | 行覆盖率 >= 80%，分支覆盖率 >= 70% |
| 重点覆盖 | 表单校验逻辑、参数名正则校验、文件类型/大小校验、OpenAPI 解析器、目录树 CRUD 逻辑、报告生成逻辑 |

前端单元测试重点：
- 用例创建/编辑表单的字段校验
- 接口参数名称正则校验 `^[a-zA-Z_][a-zA-Z0-9_]*$`
- 文件上传组件的类型和大小限制
- 目录树组件的展开/折叠/拖拽行为
- 代码编辑器的语法高亮渲染

后端单元测试重点：
- Excel 解析与逐行校验逻辑
- OpenAPI 3.x JSON/YAML 解析器
- 脚本沙箱执行与超时控制
- 报告生成（PDF/Excel）逻辑
- 目录树级联删除逻辑

### 2.2 集成测试

| 维度 | 说明 |
|------|------|
| 框架 | testcontainers-go |
| 数据库 | PostgreSQL 15（testcontainers 启动） |
| 文件存储 | MinIO（testcontainers 启动，模拟对象存储） |
| 沙箱运行时 | Docker-in-Docker（testcontainers 启动） |

集成测试重点场景：
- 用例 CRUD 全链路（API -> DB -> 响应）
- Excel 批量导入全链路（上传 -> 解析 -> 入库 -> 错误反馈）
- 接口录入 -> 在线调试全链路（录入 -> 代理请求 -> 响应展示）
- 脚本上传 -> 在线调试全链路（上传 -> 沙箱执行 -> 输出返回）
- 自动化执行全链路（选择用例+脚本 -> 执行 -> 报告生成 -> 下载）
- 目录树级联删除（删除非空目录 -> 子节点全部清理）

### 2.3 E2E 测试

| 维度 | 说明 |
|------|------|
| 框架 | Playwright |
| 浏览器 | Chromium (latest), Firefox (latest) |
| 关键路径数量 | 6 条 |

关键 E2E 路径：
1. 创建目录 -> 手动创建用例 -> 编辑用例 -> 删除用例
2. 下载 Excel 模板 -> 填写 -> 批量导入 -> 查看导入结果（含部分失败）
3. OpenAPI 导入接口 -> 表单录入接口 -> 在线调试 -> 查看响应
4. 上传 Python 脚本 -> 在线编辑 -> 调试运行 -> 查看输出
5. 选择用例+脚本 -> 一键执行 -> 查看报告 -> 下载报告
6. 删除非空目录 -> 确认级联删除 -> 验证子节点清理

## 4. 测试数据

### 4.1 数据准备

| 数据类型 | 说明 | 来源 |
|----------|------|------|
| 测试用例种子数据 | 50 条预置用例，覆盖不同目录层级 | SQL seed 脚本 |
| Excel 导入模板 | 合法模板 + 5 种异常模板（缺列、类型错误、空行、超大文件、混合正确/错误行） | fixtures 目录 |
| OpenAPI 定义文件 | 合法 YAML/JSON 各 1 份 + 非法格式 3 份 | fixtures 目录 |
| 测试脚本文件 | Go 脚本 2 个、Python 脚本 2 个、非法文件（.js/.rb）2 个、超大文件 1 个 | fixtures 目录 |
| 接口参数数据 | 覆盖所有参数类型（string/bool/int/long/float/object/array<*>）的参数组合 | 测试代码内构造 |
| 目录树数据 | 10 级嵌套目录、空目录、非空目录 | SQL seed 脚本 |

### 4.2 数据清理方案

| 策略 | 适用范围 |
|------|----------|
| 事务回滚 | 单元测试 — 每个测试用例在事务中执行，结束后回滚 |
| testcontainers 生命周期 | 集成测试 — 每个测试套件启动独立容器，结束后销毁 |
| 数据库 truncate | E2E 测试 — 每个测试场景执行前 truncate 所有业务表并重新 seed |
| 文件存储清理 | 每个测试套件结束后清空 MinIO 测试 bucket |

### 4.3 测试数据隔离

- 单元测试：mock 所有外部依赖，数据完全内存化
- 集成测试：每个测试套件使用独立的 testcontainers 实例，数据库和文件存储物理隔离
- E2E 测试：使用专用测试环境，通过唯一前缀（如 `e2e_{timestamp}_`）隔离测试数据
